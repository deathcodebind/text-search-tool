# 全文搜索竞标助手 - 立项文档草案（MVP）

## 1. 项目背景与目标

### 1.1 一句话定位
面向公开竞标信息的整理与筛选工具，通过内部全文搜索能力帮助用户快速定位可承接标的。

### 1.2 MVP 核心痛点
公开竞标信息分散、检索效率低，人工筛选成本高；需要通过关键词规则组合快速筛出可参与竞标。

### 1.3 交付策略
当前不设固定交付日期，先完成可落地使用的 MVP 并投入实际测试，再根据反馈迭代优化功能与体验。

## 2. 用户与使用场景

### 2.1 目标用户
对竞标有兴趣的用户，可按行业与能力范围筛选可承接标的。

### 2.2 典型流程
1. 用户选择目标网站（MVP 仅支持 jxemall.com）。
2. 用户提供目标网站登录信息（需具备合规企业用户身份）。
3. 用户使用目标网站自带筛选项进行第一轮筛选。
4. 系统执行拉取并入库，用户等待拉取完成。
5. 用户应用关键词规则进行第二轮全文查询。
6. 用户查看结果详情并决定是否参与竞标。

### 2.3 角色与部署形态
- MVP 为本地桌面端形态，不区分多角色权限。
- 当前聚焦单用户本地使用。

## 3. MVP 功能范围

### 3.1 必做功能
- 基于 Tantivy 的全文检索服务。
- 多关键词、多层级规则、可配置多分数综合排序。
- 中文分词能力，对接 jieba-rs（可评估 tantivy-jieba）。
- 关键词规则支持 Bool Query 核心子集（首版）。
- Tauri 桌面端打包与完整 GUI 交互流程。
- 多 crates 工程化拆分：storage、search-engine、gui、crawler（名称可调整）。

### 3.2 关键词规则（首版边界）
首版实现 Bool Query 核心子集，后续根据测试反馈决定是否在 V1 扩展到完整能力。

建议核心能力集合如下：
- must（必须命中）
- should（可选命中）
- must_not（禁止命中）
- minimum_should_match（最少 should 命中数）
- 关键词权重（强关键词/弱关键词）

### 3.3 结果展示字段（初版）
- 标题
- 发布时间
- 地区
- 来源链接
- 命中词/命中摘要

字段可根据客户反馈增删。

### 3.4 明确延期到后续版本（暂定）
- 完整 Bool Query 全量语法（若超出首版排期）。
- 多用户/多角色权限系统。
- 云端同步与团队协作。

## 4. 数据来源与处理策略

### 4.1 数据来源
- 目标网站：jxemall.com（MVP 唯一来源）。
- 抓取前提：用户需提供合规登录身份。

### 4.2 抓取频率
- 每天固定时间执行抓取。
- 时间窗口：从“前一次抓取时刻”到“当前抓取时刻”的增量数据。

### 4.3 抓取字段
- 字段清单按网站结构在 crawler crate 实施阶段逐站点梳理。
- 当前先预留可扩展字段模型。

### 4.4 去重策略
- 主去重键：来源数据 ID（权威标识）。
- 辅助去重键：目标 URL。

### 4.5 数据过期策略
- 按用户要求执行 3 天周期的数据过期清理。

### 4.6 凭证存储
- 倾向采用系统 Keychain。
- 或采用浏览器风格会话存储模型（结合 Tauri 能力评估）。

## 5. 技术架构（草案）

### 5.1 Crates 拆分（候选）
- crates/storage：本地数据库访问、去重、过期清理。
- crates/search-engine：索引构建、检索、评分、分词集成。
- crates/crawler：站点登录态处理、数据抓取、字段映射。
- crates/gui：Tauri 应用与前端交互。
- crates/shared（可选）：跨 crate 的通用模型和错误定义。

### 5.2 数据存储
- Tantivy：全文索引与检索。
- 本地数据库：SQLite（承载元数据、任务状态、规则配置、过期管理）。
- 过期清理策略：基于 expires_at 索引的定时批处理删除；与 Tantivy 索引删除联动。

### 5.3 GUI 技术栈（当前倾向）
- 前端技术栈定为 Vue 3。
- 状态管理优先采用 Vue Store（Pinia），用于承载较重的数据读取与跨页面状态共享。
- Svelte 与 React 不作为当前 MVP 主选型。

## 6. 查询与评分策略（首版）

建议首版采用可配置的线性组合评分模型：

Score = a * keyword_score + b * field_weight_score + c * freshness_score

其中：
- keyword_score：关键词命中与权重贡献。
- field_weight_score：标题/正文等字段权重贡献。
- freshness_score：时间衰减项。
- a/b/c：可配置权重参数。

## 7. 合规与风险（初版）

### 7.1 合规约束
- 仅抓取具备合法访问权限的公开竞标数据页面。
- 登录行为与数据访问需符合目标网站条款。

### 7.2 主要风险
- 目标站点结构变更导致抓取失效。
- 登录流程（验证码/风控）导致自动化不稳定。
- 中文分词词典与行业术语适配不足导致召回/精度波动。

### 7.3 风险缓解方向
- crawler 站点适配器化设计。
- 抓取任务失败重试与告警日志。
- 分词词典可热更新（后续版本）。

## 8. 里程碑（草案）

### M0：技术验证
- Tantivy + 中文分词可用性验证。
- jxemall.com 登录与增量抓取打通。

### M1：MVP 可用版
- 端到端流程可跑通（抓取 -> 入库 -> 索引 -> 查询 -> 展示）。
- 核心规则与评分可配置。
- Tauri 桌面端可安装运行。
- 主要交互流程无明显错误（以实际试用为准）。

### M2：试用迭代
- 基于真实用户反馈优化规则能力与展示字段。
- 修复稳定性问题，补齐关键测试。

## 9. 待确认事项（需要补充）

- Vue 3 状态管理细节（Pinia 模块划分、缓存与失效策略）。
- 首版性能压测规模（索引量级、并发规模）与执行时点。
- 日志与可观测性方案（本地日志等级、故障排查信息）。
- 打包发布策略（已确认 Windows + macOS 双平台首发）。
- 成功指标（MVP 阶段可先以流程可用为主）。

数据库选型结论：
- MVP 阶段确定使用 SQLite。
- RocksDB 作为后续规模化备选，不在当前首发范围。

补充说明：
- 查询延迟目标暂定为 300ms 以内。
- 性能测试优先级靠后，计划在功能按用户反馈完成主要调整后执行。
- MVP 达成标准：完整用户流程可跑通，且无明显错误。
